Overview

The JavaScript controls are intended for advanced HTML authors who are not afraid to tinker with HTML and modify JavaScript variables. If the previous sentence didn't make sense to you, you may want to make a pot of coffee. There is a "newbie" section included in this documentation, which attempts to help explain some of the techno mumbo-jumbo it's the best I can offer without actually showing up at your office and holding your hand.

Basically the way it works is that the wimpy.js file contains a series of functions that create HTML code on the fly, and use the JavaScript Kit developed by Macromedia to allow JavaScript to communicate with the internal Actionscript contained within the Wimpy MP3 Player.  The end result is that JavaScript can control Wimpy MP3 Player.

The JavaScript Controls consists of 3 files:

wimpy.swf
wimpy.js
some_page.html

<- The player
<- The wiring
<- Your HTML page

The remainder of this documentation explains how to implement the JavaScript Controls. I recommend taking a look at the source HTML code in the example HTML pages to see how the code is implemented in a **real** HTML page. The examples should work on your local PC, but you may experience problems when viewing locally depending on the security settings you have established through the Flash plugin installed in your browser. The best way to test things out is to upload all the files contained in this package to your web server.

makeWimpyPlayer()

Initializes the JavaScript Controls and renders a player on your page based on the options established within the wimpy.js file. The wimpy.js file contains more information about the available options in the javaScript comments.

This is required to use any of the JavaScript Controls. In order to take advantage of the JavaScript Controls, an instance of the player must  be established through javascript using makeWimpyPlayer.

Only one player can be rendered and controlled on a page.

To render a player:

<script language="JavaScript" >
  makeWimpyPlayer();
</script>

To render a player with a default MP3 file:

<script language="JavaScript" >
  makeWimpyPlayer("file1.mp3");
</script>

To render a player with a few files, seperate each file with a "pipe" character:

<script language="JavaScript" >
  makeWimpyPlayer("file1.mp3|file.mp3|file3.mp3");
</script>

wimpySwf = "http://www/path/to/wimpy.swf";

trackPlays = "";

URL to default iamge (jpg or swf format only)

This will cause the player to "wait" the specified period of time (in seconds) before the track actually starts to play. This option is good for slow internet connections, since it allows the file to load X number of seconds before it starts to play -- thereby minimizing a "choppy" playback. Set this to the type of file it should load.

js_wimpy_returnOnComplete

Specifies a function that Wimpy should ping when:

- Each time a track begins to play.
- When each track finishes playing.
- The function "wimpy_getTrackInfo" is initiated.
- The function "wimpy_returnPlaylistItem" is initiated.

If set, Wimpy will "ping" the function name defined for this variable.

Example:

js_wimpy_returnOnComplete = "myFunction";

funtion myFunction(WimpyReturnArgs){
   alert ("wimpy returns:"+ WimpyReturnArgs);
}

In the example above, Wimpy will initiate the function "myFunction" each time a track starts to play and when the track is finished palying. (And when either wimpy_getTrackInfo or wimpy_returnPlaylistItem have been initiated. )

NOTE: If the user clicks the "pause" or "stop" button, the function will NOT be pinged. In order for this function to be pinged when the track is complete, the track must reach the end of it's play time "naturally."

 

limitPlaytime

Limits the amount of time that the file is allowed to play. Set this to a number of seconds that files should be allowed to play.

The following example above will imit play time to 30 seconds:

limitPlaytime="30";

 

startupLogo

You can use either a JPG or SWF image as the start up logo, which will replace the default "W" logo when the player starts up.

When the image is loaded, it is centered within the player's bounding box. The image is not resized, it is only centered.

Example:

startupLogo = "http://www.yoursite.com/images/logo.jpg";

NOTE: If you are using an SWF logo, Wimpy will center the loaded movie based on the dimensions of all of the objects on the main stage, so if there are any "invisible" items, or if objects "hang off the edge" of the main stage, or if the objects are smaller than the main stage, the loaded logo may not get centered properly. Wimpy can not determine the size of the main stage, it can only determine the size of all the objects on the main stage.

http://www.yoursite.com/bob's files/bob.jpg
... the only way to ensure that the apostrophe would work would be to wrap the URL with double quotes. if we wrapped the URL with single quotes, then the code would "breaK" because the apostrophe would act as the last (or ending) quote:

Bad:
'http://www.yoursite.com/bob's files/bob.jpg'
Good:
"http://www.yoursite.com/bob's files/bob.jpg"

Likewise, if we wanted to use a double quote in the URL, we would have to wrap the URL with single quotes:

Bad:
"http://www.yoursite.com/bob said "hi"/ hello.jpg"
Good:
'http://www.yoursite.com/bob said "hi"/ hello.jpg'

I do not recommend ever using weirdo characters in any file name or folder name. Try and use only alpha-numeric characters.

The reason I went off on this long explanation is because sooner or later you are going to want to use a file with an apostrophe in your wasp code -- like when you try and reference an FLV video such as:

waspPopup("bob's video.flv", 320, 240)

It will invariably happen, and now you know why.

2.5 Understanding the available "methods" contained in the JavaScript Controls.

There are a number of "methods" of "functions" you can call in your HTML pages. One being a call to embed a player directly into the page, this method is required since it "initializes" the javaScript Controls. The remainder of the the functions take advantage of standard HTML hyperlink tags to control the player.

2.6 Establishing a player.

In order to use the JavaScript Controls, we must first establish a Wimpy MP3 Player on our page. By doing so, we "initialize" the player. This just means that we are establishing the JavaScript code and the player so that the browser will know how to pass the information between to and from the player. One reason that we need to establish a player is because within the HTML we need to identify a "name" and "id" for the player. The "name" and "id" act as a label or "name tag" so that the browser can identify the player "by name" so that when a JavaScript Control function is actuated, the browser will say to it's self "to whom shall I give this command" -- if the player and JavaScript Controls did not have a reference name, the browser will not know who to give the instructions to. But since we are "initializing" the JavaScript controls and providing the Wimpy MP3 Player with a name, then the JavaScript controls can tell the browser to "give this instruction to the Wimpy MP3 Player that has this ____ name.

Another side benefit to using JavaScript to establish a player is that we can do the same things that the Customizer tool does -- create a bunch of HTML code -- without having to continually use the Customizer tool. The "makeWimpyPlayer" function allows us to render all the HTML required to display the player without all the overhead and complexity of the HTML that the Customizer tool outputs.

To establish a player in your HTML page include the following code anywhere between the opening <body> and closing </body. tags in your HTML page. This function will write all the HTML code necessary to display the player based on the settings defined in the wimpy.js file.

Example:

<script>
  makeWimpyPlayer();
</script>

 

2.7 Controlling the player.

The player is controlled by taking advantage of standard <a href> hyperlink tags. e.g. a "standard link" HTML code. With a standard HTML hyper link, we usually just reference a URL in the "href" attribute of a hyperlink tag.

Example of a standard HTML hyperlink tag:

<a href="URL"> Click Here </a>

The <a> tag contains one attribute named href (which stands for "Hyperlink Reference"). The <a> tag in our standard HTML example above is wrapped around the words "Click Here" -- so that when the words "Click Here" are clicked, the browser knows to go to a new page... Or, the browser "hyperlinks" to a new page.

With JavaScript we can take advantage of the simplicity of the <a> hyperlink tag by adding an additional attribute to the <a> hyperlink tag. The additional reference is an "onClick" attribute.

Example:

<a href="javascript:;" onClick="do something" > Click Here </a>

In order to tell the browser that we are using JavaScript instead of a traditional hyperlink reference, we replace the usual URL with the word "javascript" followed by a colon and semi-colon. Doing so indicates to the browser that we want to over-ride the traditional hyperlink action with a javascript function.

The additional attribute named "onClick" tells the browser what kind of user interactivity we want the browser to react to. There are a variety of interactive options we can set this attribute to, but onClick is the most widely used option -- supported by all browsers, so we'll just stick with this option.

So, now that we have told the browser to deal with this hyperlink when the link is clicked on -- or "onClick"  -- we need to inform the browser which JavaScript function to use when the link is clicked.

In the same respect that the traditional href attribute "points" the browser to a new URL, with the onClick attribute, we are "pointing" the browser to a JavaScript function. The JavaScript function is actually contained in the wimpy.js file, which we referenced in the <head> tag... here we are just "calling" the function with the specifics about how we want to control the player.

The wimpy_loadAndPlay function allows you to include addition information about the file that you are loading into the player. information such as Artist name, Track Title, A URL to the cover art image and a hyper link that the cover art image will use when the user clicks on the image.

The way that the function puts this information into the player is by the references we use within the parenthesis. As you may have noticed, all function calls contain a set of parenthesis after the name of the function (). These parenthesis allow us to send dynamic information into the player so that we can re-use the function with different information.

Think of a function as though it were a coffee grinder. you put beans into the top of the coffee grinder, the gears inside the grinder grind up the coffee , and the end result is usable coffee. A function is very similar. stuff gets "put into" the function, the function processes the information, then outputs usable code, or performs a specific task. The stuff between the parentheses is like the area that accepts the raw coffee beans and the result is the usable coffee.

If you were to open the wimpy.js file in a text editor and find the reference to the wimpy_loadAndPlay function, you will notice that there are words inside the parentheses that refer to specific "kinds" of information. When we access the function, we enter "real" information in the same place that the "holder" reference words are located. Each different kind of information that we put into the function is separated by a comma. In the wimpy.js file, each item is just a reference word, so the reference word (or variable) does not have to have quotes around it. however, when we attempt to use the function "in real life" we must wrap each piece of information we send into the function with quotes so that the function will recognize the data as a text "string" -- otherwise JavaScript will attempt to interpret the stuff we put into the function as other reference variables. By wrapping the stuff with quotes we are basically sending in "raw" data (or full beans).

It is also important to maintain the proper order of the stuff that gets sent in. In other words, if a function accepts 3 items.

my_function(x, y, z)

Then we must include three items when we call the function -- and the reference to each must be in the same order.

For example, lets say we have a function named "make_a_sentance" that accepts three kinds of information:

1. A persons name
2. A day
3. A location

The function could look something like:

function make_a_sentance(theName,theLocation, theDay){
   document.write(theName + " went to " + theLocation + " on " + ".");
}

When we call the function we would use something like:

make_a_sentance("Bob" , "the store", "3/14/2006")

This function would output the following:

Bob went to the store on 3/14/2006.

If, when calling the function, we put the information in the wrong "slot" the sentence would not print out properly. So we must ensure that we put the proper kind of information into the proper "slot" so that the function can put the stuff in the right spot. If you don't have anything for a specific "slot" then you would use empty quotes, but you still have to put "something" into the "slot" -- even if it's nothing.

Example:

make_a_sentance("Bob" , "", "3/14/2006")

This sentence would print out as:

Bob went to   on 3/14/2006.

But if we simply omitted or didn't reference the second "slot" at all the function would output:

Bob went to 3/14/2006 on  .

Because the function is thinking that the seconds "slot" should be a reference to a location, not a date.

So in the same respect, when using the JavaScript controls, you need to reference the proper kind of data in the proper slot. If you do not want to include anything for that particular kind of data, say, you didn't want to include an image or artist name, then you would simply use empty quotes.
